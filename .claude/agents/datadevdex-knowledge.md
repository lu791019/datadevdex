## [流程] implementation_plan.md 的決策紀錄補強

**來源**：Dex 確認 / 2026-03-26
**觀點**：規劃文件不只記「做什麼」，也要記「不做什麼」和「為什麼不走別條路」
**細節**：
- 現有區塊：Goal Description / User Review Required / Proposed Changes / Verification Plan
- 新增 `Non-goals`：放在 Goal Description 之後，列出「看起來該做但刻意不做」的項目 + 一句理由
- 新增 `Alternatives Considered`：放在 Proposed Changes 之後，每個替代方案一行描述 + 為什麼不選
**脈絡**：Solo developer 三個月後常忘記當初的決策邊界和替代方案，這兩個區塊成本低但能大幅提升決策可追溯性

## [流程] ADR 機制與觸發條件

**來源**：Dex 同意建立 / 2026-03-26
**觀點**：ADR 只用在架構級決策，不是每個 feature 都寫
**細節**：
- 位置：`docs/adr/`，輕量格式（標題 + 狀態 + 脈絡 + 決策 + 後果）
- 觸發條件：跨模組架構影響、引入新依賴、改變資料流
- 預期頻率：每月 1-2 篇
- 日常決策由 implementation_plan.md 的 Alternatives Considered 覆蓋，不需要 ADR
**脈絡**：避免 ADR 變成負擔，保持輕量但在關鍵時刻留下紀錄

## [流程] /tech-lead challenge 使用時機

**來源**：Dex 確認 / 2026-03-26
**觀點**：Challenge 模式不是常規流程，只在大架構變更時才有價值
**細節**：
- 適用：新建子系統、改變核心資料流、引入重大依賴
- 不適用：一般 Phase 開發、feature 實作、bug fix
- 原因：Solo developer 的時間有限，Devil's Advocate 在小決策上 ROI 不高
**脈絡**：與其每次都跑 challenge，不如靠 Non-goals + Alternatives Considered 做輕量自我檢視

## [觀念] 「做」vs「做好」— 完成的定義升級

**來源**：Dex 分享文章 / 2026-03-26
**觀點**：「做完」和「做好」的差距是工程紀律和成果追蹤
**細節**：
- 做 = 功能可運作；做好 = 有測試、有監控、不會再發同樣問題
- 無聊但必須做的事是自動化候選 — 因為做過所以有資格設計更好的抽象（Replit CEO）
- Scope 先行：先展現更大範疇的影響力，升遷才會跟上；但不等於什麼都自己做，要會 delegate
- 思考框架是「創造價值」而非「完成任務」— 測試和監控就不再是額外負擔
- 先決條件：釐清當前脈絡下什麼是重要的（新創 vs 大廠的「重要」不同）
**脈絡**：Dex 的 feedback loop 已解決 code 層面的「做好」，缺的是功能/Phase 層面的成果追蹤

## [觀念] 成果追蹤 — 做完不等於有價值

**來源**：Dex 分享文章（做 vs 做好）/ 2026-03-26
**觀點**：開發了但沒累積出成果，從結果來看與沒開發無異
**細節**：
- 很多工程師認為商業成果是 PM 的事，但公司聘你是為了透過產出提高收益
- 對 Solo developer 更關鍵：沒有 PM 替你追蹤，自己不追蹤就沒人追蹤
- 建議：Phase merge 後在 PLAN.md 加 `## Phase Retrospective`（使用頻率 / 解決的痛點 / 重來會怎麼做）
**脈絡**：與前兩篇結論串連 — 決策留痕（第二篇）+ 結果留痕（第三篇）= 完整閉環

## [觀念] 無聊的事是自動化信號

**來源**：Dex 分享文章（做 vs 做好，引 Replit CEO）/ 2026-03-26
**觀點**：無聊但必須做的事，代表有機會被好的抽象化與自動化解決
**細節**：
- 多做無聊的事 → 對它有更深洞見 → 有機會提出全新解法
- dex-agent-os 的 `/daily-all`（14 步 → 一鍵）就是這個原則的實踐
- 不是為了偷懶，而是因為做過所以能設計好的自動化
**脈絡**：驗證了 Dex 建構 dex-agent-os 的核心動機
